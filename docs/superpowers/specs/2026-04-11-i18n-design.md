# Internationalization (i18n) Design

## Overview

Add multi-language support to the Tarot app using `next-intl` with URL-based routing (`/en/`, `/no/`, `/ru/`). Translates all UI strings, card names, readings, subscription plans, and legal pages. Language is auto-detected from browser, with a manual dropdown override in the header.

## Supported Locales

- `en` — English (default, fallback)
- `no` — Norwegian (Bokmal)
- `ru` — Russian

Adding a new language = adding a new folder under `messages/` and registering the locale in `routing.ts`.

## Library

`next-intl` — the standard i18n library for Next.js App Router. Handles:
- URL prefix routing and middleware
- Browser language detection via `Accept-Language`
- Cookie-based locale persistence
- `useTranslations` hook for components
- Server component support via `getTranslations`

## File Structure

```
messages/
  en/
    ui.json          # UI strings (buttons, titles, labels, headings)
    cards.json       # 78 card names keyed by card id
    readings.json    # Readings: meaning, keywords, past, present, future per card
    legal.json       # Terms of Service + Privacy Policy full content
    plans.json       # Plan names, features, CTA labels
  no/
    ui.json
    cards.json
    readings.json
    legal.json
    plans.json
  ru/
    ui.json
    cards.json
    readings.json
    legal.json
    plans.json

src/
  i18n/
    routing.ts       # defineRouting config (locales, defaultLocale)
    request.ts       # getRequestConfig — loads & merges JSON files per locale
    navigation.ts    # createNavigation — locale-aware Link, useRouter, usePathname
  middleware.ts      # next-intl middleware for detection + URL rewriting
  app/
    [locale]/        # All pages move under dynamic locale segment
      layout.tsx     # Root layout with NextIntlClientProvider
      page.tsx
      subscription/page.tsx
      terms/page.tsx
      privacy/page.tsx
    api/             # API routes stay outside [locale] — no translation needed
```

## Routing & Locale Detection

### Middleware (`src/middleware.ts`)

Uses `next-intl/middleware` with the routing config. Detection priority:
1. URL prefix (`/no/subscription`)
2. Cookie (set when user picks language from dropdown)
3. `Accept-Language` browser header
4. Falls back to `en`

API routes (`/api/*`), static assets (`/_next/*`, `/_vercel/*`), and files with extensions are excluded via matcher.

### URL Structure

- `/en/` — English
- `/no/` — Norwegian
- `/ru/` — Russian
- Default locale shows prefix for consistency
- Root `/` redirects to detected locale

### Card Names (`cards.json`)

Keyed by card id. The `name` field is removed from `data.ts`.

```json
{
  "arcana-0": "The Fool",
  "arcana-1": "The Magician"
}
```

### Readings (`readings.json`)

Keyed by card id, nested with meaning/keywords/past/present/future. Replaces `tarotReadings.ts`.

```json
{
  "arcana-0": {
    "keywords": ["beginnings", "innocence", "spontaneity", "free spirit"],
    "meaning": "The Fool speaks of new beginnings...",
    "past": "A bold leap of faith...",
    "present": "You are being called...",
    "future": "A thrilling new chapter..."
  }
}
```

### Plans (`plans.json`)

Plan names and feature lists. `plans.ts` keeps `id`, `priceLabel`, `interval` (non-translatable config).

```json
{
  "FREE": {
    "name": "Free",
    "features": ["3 readings per day", "Resets at midnight"]
  },
  "MONTHLY": {
    "name": "Monthly",
    "features": ["Unlimited readings", "Reading history", "..."]
  }
}
```

### Legal (`legal.json`)

Full Terms of Service and Privacy Policy content, structured by section:

```json
{
  "terms": {
    "title": "Terms of Service",
    "lastUpdated": "Last updated: April 6, 2026",
    "intro": "Welcome to Tarot...",
    "sections": {
      "entertainment": { "title": "1. Entertainment Only", "content": "..." },
      "eligibility": { "title": "2. Eligibility", "content": "..." }
    }
  },
  "privacy": {
    "title": "Privacy Policy",
    "lastUpdated": "Last updated: April 6, 2026",
    "intro": "This Privacy Policy explains...",
    "sections": {
      "dataController": { "title": "1. Data Controller", "content": "..." }
    }
  }
}
```

## Language Switcher

A dropdown added to the `Header` component, next to the main menu. Shows language names in their native form:
- English
- Norsk
- Pyccкий

Uses `next-intl`'s `useRouter` and `usePathname` to switch locale while staying on the current page. The middleware's built-in cookie persists the choice.

## Components Requiring Translation

Every component with hardcoded English text gets `useTranslations`:

- `OfferBlock` — "Shake the Deck", "Join the Circle", "Discover Your Fate"
- `Tarot` — "Unveil Your Destiny", "Revoke and Retry", "The Cards Have Spoken"
- `MainMenu` — "Pricing", "Reveal Yourself", "Welcome, ..."
- `Header` — language dropdown
- `LoginForm` — form labels, buttons, error messages
- `UserProfile` — field labels, "Upgrade"
- `SubscriptionPlans` — title, subtitle, plan display
- `Footer` — "Terms of Service", "Privacy Policy"
- `PageShell` — modal titles
- Terms page — full content from `legal.json`
- Privacy page — full content from `legal.json`

## Data File Changes

- **`data.ts`** — remove `name` field from card objects (names come from `cards.json`)
- **`tarotReadings.ts`** — deleted entirely, replaced by `readings.json` per locale
- **`plans.ts`** — keep `id`, `priceLabel`, `interval`. Remove `name` and `features` (come from `plans.json`)

## Navigation Changes

All `Link` imports from `next/link` and `useRouter` from `next/navigation` switch to the locale-aware versions from `@/i18n/navigation`. This ensures all internal links automatically include the locale prefix.

## What Does NOT Change

- API routes — no translation, stay at `/api/`
- Database schema — no changes
- Auth flow — unchanged
- Card images — language-independent
- Price labels — stay in `plans.ts` (currency formatting is a separate concern)

## Fallback Behavior

If a translation key is missing in the current locale, `next-intl` falls back to the English (`en`) message. This is configured in `request.ts` by merging the English messages as a base.

## SEO

- Each locale gets its own URL (`/en/terms`, `/no/terms`, `/ru/terms`)
- `hreflang` tags added to `<head>` via layout metadata
- `<html lang="...">` set dynamically from the locale param

## Translation Generation

Norwegian and Russian translations will be generated for all content as part of implementation. The user will review and correct them after.
